# 🛡️ День 13: LLM Security Gateway Proxy

## Обзор
В этом проекте реализован LLM Security Gateway — обратный прокси между пользователями и LLM-провайдером (совместимым с Ollama/OpenAI). Приложение обеспечивает безопасность входящих и исходящих данных, перехватывая запросы, проверяя их на наличие конфиденциальной информации (Input Guard) и валидируя сгенерированные ответы (Output Guard), а также отслеживая использование токенов и стоимость.

Приложение построено на чистом стеке **Kotlin** с использованием **Ktor** (сервер и клиент) и применяет **langchain4j** для токенизации и вспомогательных утилит.

## Основные возможности
- **Прозрачное проксирование:** Перенаправляет стандартные JSON-запросы эндпоинта `/v1/chat/completions` напрямую в локальный инстанс Ollama.
- **Input Guard (Защита ввода и поиск секретов):**
  - Использует регулярные выражения (Regex) для обнаружения API-ключей (AWS, GitHub, OpenAI), email-адресов, номеров кредитных карт и телефонов.
  - Поддерживает настраиваемые политики применения: `REDACT` (маскирует секреты, например `[REDACTED_API_KEY]`) или `BLOCK` (полностью отклоняет запрос).
- **Output Guard (Безопасная генерация):**
  - Сканирует обычные и потоковые ответы на наличие сгенерированных моделью секретов (галлюцинаций), подозрительных URL-адресов и shell-команд (например, `rm -rf`, `curl`).
  - Реализует **буфер скользящего окна (Sliding Window)** для потоковых ответов (SSE). Он накапливает частичные чанки для точного определения разорванных на части запрещенных паттернов (например, `'c'` + `'url'`) и немедленно обрывает поток при обнаружении нарушения.
- **Отслеживание стоимости и Rate Limiting:**
  - Извлекает объект `usage` из ответов LLM для отслеживания расхода токенов.
  - Включает резервный механизм локального подсчета токенов для стриминга или случаев, когда данные `usage` отсутствуют в ответе провайдера.
  - Базовое ограничение частоты запросов в оперативной памяти (in-memory rate limiting) для предотвращения злоупотреблений.
- **Аудит и логирование:** Логирует все запросы, обнаруженные нарушения, маскирование данных и метрики использования в консоль и в файл.

## Технологический стек
- **Kotlin** (Чистый Kotlin, без Spring Boot)
- **Ktor** (Сервер и клиент с использованием Coroutines и Flows)
- **Langchain4j** (Утилиты для токенизации)
- **SLF4J / Logback** (Аудит и логирование)

## Как запустить
1. Убедитесь, что Ollama запущена локально с нужной моделью (например, `llama3.1:8b`).
2. Соберите и запустите Ktor-приложение: `./gradlew run`.
3. Отправьте запросы через `curl` на `http://localhost:8080/v1/chat/completions`.
